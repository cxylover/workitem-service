# 过程记录

## 1. 需求理解

### 核心场景

建设一个轻量级的 AI 辅助研发工作项流转与需求澄清系统，面向研发团队内部使用。

### 核心对象

1. **工作项** - 需求、缺陷或任务的载体
2. **澄清问题** - 需求不明确处的问题记录
3. **状态历史** - 工作项状态变更的审计日志
4. **AI 分析结果** - AI 辅助分析的结构化输出

### 核心业务规则

1. 工作项状态按照预定义的规则流转
2. 高优先级未解决的澄清问题会阻断状态流转
3. 状态变更必须记录历史
4. AI 分析结果需要结构化返回

## 3. 技术方案

### 技术栈选择

- **Spring Boot 3.2.5**: 最新稳定版本，性能优秀
- **H2 数据库**: 内嵌数据库，无需额外安装，便于演示
- **原生 HTML/JS**: 无需构建工具，快速实现前端页面

### 工程结构

```
com.assessment.workitem
├── controller    # REST 控制器
├── service       # 业务逻辑
├── repository    # 数据访问
├── model
│   ├── entity    # 实体类
│   └── enums     # 枚举类
├── dto           # 数据传输对象
├── exception     # 异常处理
└── config        # 配置类
```

### 关键设计决策

1. **状态机实现**: 使用枚举 + Map 定义合法流转规则，便于维护和扩展
2. **阻断规则**: 在状态流转时检查澄清问题，而不是在创建问题时
3. **AI 服务接口化**: 定义接口便于后续替换为真实 LLM 实现
4. **统一异常处理**: 使用 @RestControllerAdvice 统一处理异常

## 4. API / 数据 / 状态设计

### 状态流转设计

```
DRAFT → ANALYZING → READY → IN_PROGRESS → TESTING → COMPLETED
```

- **DRAFT**: 初始状态，工作项刚创建
- **ANALYZING**: 需求分析阶段
- **READY**: 准备就绪，可以开始开发
- **IN_PROGRESS**: 开发中
- **TESTING**: 测试中
- **COMPLETED**: 已完成（终态）

### 数据模型设计

1. **WorkItem**: 核心实体，包含标题、描述、类型、优先级、状态等
2. **ClarificationQuestion**: 澄清问题，关联到工作项
3. **StatusHistory**: 状态变更历史，记录每次流转
4. **AiAnalysisResult**: AI 分析结果，JSON 格式存储

### API 设计原则

- RESTful 风格
- 资源路径清晰
- 统一响应格式
- 明确的错误码和错误信息

## 6. 遇到的问题

### 问题一：状态流转规则设计

**问题**: 如何设计灵活且可维护的状态流转规则？

**解决方案**: 使用枚举 + Map 定义合法流转规则，集中管理，便于扩展。

### 问题二：澄清问题阻断时机

**问题**: 何时检查澄清问题的阻断规则？

**解决方案**: 在状态流转时检查，而不是在创建问题时，这样更符合业务逻辑。

### 问题三：AI 分析结果格式

**问题**: 如何设计结构化的 AI 分析结果？

**解决方案**: 使用 JSON 格式存储，不同分析类型有不同的结构，便于前端解析和展示。
